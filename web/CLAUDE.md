# web/ (multi-user web app)

A web frontend so friends can use the downloader without setting up Python/ffmpeg/Spotify credentials themselves. Scope is deliberately narrow: **friends/invite-only, not public** тАФ running a public YouTube-audio-ripping service for arbitrary strangers is a real copyright/ToS risk (cf. youtube-mp3.org-style shutdowns); this stays small and gated. Two features are exposed: the manual artist/album download form, and (new) each friend logging into their **own** Spotify account to get a personalized scan of region-locked/missing albums, picked from `sync_missing.py`'s logic but adapted to run per-user on the server instead of as a single-user local CLI script.

**Frontend is now React + TypeScript + Vite** (`web/frontend/`), not a single Jinja2 page тАФ the site grew from one static form into a real multi-view app (connect account тЖТ scan тЖТ review results тЖТ track several downloads at once), which is what justified the framework. FastAPI serves the built SPA and the API from the same origin.

**Site access vs. Spotify login тАФ two different things.** There's still no site-wide passcode; access control is just not sharing the Render URL with anyone outside the friend group (see `.env`'s lack of `SITE_USERNAME`/`SITE_PASSWORD` тАФ that was removed on purpose). Separately, and unrelated to site access, a friend can click "Connect Spotify" to log into their **own** Spotify account (Authorization Code flow, real user login + scopes) so the scan feature can read their saved albums/playlists тАФ this is per-user identity for a feature, not a gate on the site itself. The manual download form works with no login at all, exactly as before.

**Run locally:**
```
.venv\Scripts\uvicorn web.app:app --reload --port 8000
```
plus, in a second terminal, either:
- `cd web/frontend && npm install && npm run dev` тАФ Vite dev server with hot reload, proxying `/login`, `/scan`, `/jobs`, `/status`, etc. back to the FastAPI server on :8000 (see `web/frontend/vite.config.ts`), or
- `cd web/frontend && npm install && npm run build` once, then just hit `http://127.0.0.1:8000/` directly тАФ FastAPI serves the built `dist/` for everything, same as production.

**Always use `http://127.0.0.1:5173`, never `http://localhost:5173`, for the Vite dev server.** `vite.config.ts` binds explicitly to `host: '127.0.0.1'` for this reason: the Spotify OAuth session cookie is scoped to whatever host the browser actually used, but Spotify's redirect after login always lands on `127.0.0.1:8000/callback` (from `SPOTIFY_WEB_REDIRECT_URI`), bypassing the Vite dev server entirely. If the frontend were visited via a different host string (`localhost` resolves to a different address than `127.0.0.1` as far as cookie scoping is concerned, and on this project's dev machine `localhost` resolved to IPv6 `::1` while Vite wasn't listening there by default), the login cookie set during `/login` wouldn't be sent back on `/callback`, and login would silently fail with "No active login session."

Only needs the Spotify credentials (see root `CLAUDE.md`) plus `SPOTIFY_WEB_REDIRECT_URI` (see below) тАФ still no site-auth env vars.

## Architecture (all in `web/`)

- `jobs.py` тАФ in-memory `Job` registry shared by **both** download and scan jobs (`job_type: "download" | "scan"` discriminator on one dataclass тАФ the registry/TTL/eviction plumbing is identical either way), but run through **two separate** bounded queues/worker pools: `DOWNLOAD_WORKER_COUNT = 2` (ffmpeg/CPU-bound) and `SCAN_WORKER_COUNT = 1` (long-running, rate-limit-bound). Kept separate so a friend's big library scan can't head-of-line-block someone else's simple album download. **Must run with a single uvicorn worker** (`--workers 1`) тАФ job state is in-process memory. Expired jobs (`JOB_TTL_SECONDS`, 1 hour) are evicted by `cleanup_loop()`. A scan job that ends up `"rate_limited"` auto-requeues itself (same job id, so the frontend's poll of `/status/{id}` just sees it cycle back to `"queued"` тЖТ `"running"`) up to `MAX_SCAN_AUTO_RETRIES` (3) times, waiting `min(Retry-After, MAX_AUTO_RETRY_DELAY_SECONDS)` between attempts via a background `asyncio` task (not a blocking sleep, so it can't stall other friends' scans behind it in the single-worker queue) тАФ only after that's exhausted does it sit waiting for a manual "Resume scan" click, same as before.
- `downloader_adapter.py` тАФ bridges a download `Job` to `album_downloader.download_album()`: per-job temp dir, `progress_callback` writing into `Job.progress`, zips the result, deletes the uncompressed mp3s.
- `sessions.py` тАФ in-memory `Session` registry (Spotify login state per friend), mirroring `jobs.py`'s pattern: a plain dict keyed by an opaque `sid`, evicted after `SESSION_TTL_SECONDS` (6 hours тАФ deliberately longer than a download job's 1 hour, since a login needs to survive a possibly-long rate-limited scan, reviewing results, and watching several downloads finish). The client only ever holds the opaque `sid` cookie (`httponly`, `samesite=lax`, `secure` only over HTTPS) тАФ the actual Spotify token and OAuth CSRF nonce live server-side. Also caches `spotify_user_id` (populated by `/me`, same call that already fetches `display_name`) тАФ used by `scan_adapter.py` to key each friend's on-disk scan cache.
- `spotify_auth.py` тАФ per-user Spotify Authorization Code login, distinct from both `album_downloader.py`'s Client Credentials flow and `sync_missing.py`'s CLI-only `get_user_spotify_client()` (local file cache, interactive open-browser flow тАФ neither fits a server handling many different friends). `SessionCacheHandler` reads/writes a `Session`'s `token_info` so a fresh `SpotifyOAuth` built per-request still sees (and persists refreshes to) the same session's token. **Handles OAuth `state`/CSRF validation itself** тАФ spotipy's `parse_response_code()`/`get_access_token()` never check `state`, that only happens inside spotipy's interactive/local-server helpers which this app doesn't use, so without doing it manually a login-CSRF attack is possible (binding a victim's session to an attacker's Spotify account). `start_login()`/`finish_login()` generate/verify a nonce explicitly. `get_authenticated_client()` builds the client with `retries=0, status_retries=0` тАФ see the root `CLAUDE.md`'s rate-limit note for why both are needed, not just one.
- `scan_adapter.py` тАФ adapts `sync_missing.py`'s region-lock scan to run as a web `Job`, reusing `sync_missing.py`'s cache-aware gather machinery directly (`_gather_library_albums`, `_load_cache`/`_save_cache`, `_all_cached_album_ids`, `_chunked`, `_locked_keys_for_batch`, `_track_key`, `_RECOVERABLE_HTTP_STATUSES` тАФ all pure, `sp`-parameterized functions, safe across concurrent users' `sp` clients). Two caches are involved, split by privacy: each friend's own file under `WEB_CACHE_DIR` (`web/.web_scan_cache/`, gitignored), keyed by Spotify user id, holds the private "what does this friend have" data (saved albums, liked songs, playlists) тАФ this is the CLI's cache format, just one file per friend instead of the CLI's single local `CACHE_PATH`, so two friends' library contents never mix. Region-lock status is handled separately via `catalog_cache.py` (below), since it isn't private. Because the per-user cache file is durable and updated incrementally (each playlist, then each lock-check batch, saved as it's processed тАФ exactly like the CLI), it doubles as both the across-scan cache **and** the resume mechanism for an interrupted scan; there's no separate in-memory checkpoint dict to build or pass around (this replaced an earlier design where `Job.checkpoint` held `gathered_albums`/`scanned_playlist_ids`/etc. in memory only). The local-library "already downloaded" `BASE_MUSIC_PATH` check is dropped entirely (the server has no visibility into any friend's local files); market is a user-provided field, not the CLI's hardcoded `DEFAULT_MARKET`. On a 429/500/502/503/504 in either phase, the job's status becomes `"rate_limited"` (not `"error"`), both caches are left with whatever they've saved so far, and (per `jobs.py` above) it either auto-requeues itself or waits for "Resume scan" тАФ either way, `resume_scan_job()` just starts a brand-new scan `Job` for the same session; no checkpoint needs handing over since both caches already have everything.
- `catalog_cache.py` тАФ a **shared, cross-user** SQLite cache (stdlib `sqlite3`, no new dependency) of region-lock results, keyed by `(album_id, market)` тАФ deliberately *not* part of the per-user file above, because whether an album is locked in a given market is a fact about Spotify's catalog, not about any one friend's library. Once any friend's scan checks an album+market combo, every other friend's scan with an overlapping album (common for popular music) reuses that result for free instead of re-paying the 2-API-call lock-check cost тАФ a real dent in the "Spotify rate limits are shared app-wide" problem called out below. TTL and staleness rules mirror the CLI's (`LOCK_STATUS_TTL_DAYS`, imported from `sync_missing.py`, not duplicated). Lives on the same ephemeral Render disk as the per-user cache files (`web/.web_scan_cache/catalog.db`) тАФ same durability trade-off, see "Known limitations". `_connect()` callers must use `closing(_connect())` (not bare `with _connect() as conn:`) тАФ `sqlite3.Connection.__exit__` only commits/rolls back the transaction, it does **not** close the connection, so the bare form leaks a file handle per call (caught during testing as a Windows `PermissionError` when a leftover connection kept a test DB file locked).
- `app.py` тАФ routes: `GET /` + `/assets/*` (serves the built SPA via `StaticFiles`/`FileResponse` from `web/frontend/dist`; `/` is served with `Cache-Control: no-store` specifically so a browser never holds onto an old `index.html` pointing at JS/CSS asset filenames a later `npm run build` has since replaced тАФ hit this exact bug once during development, showed up as a blank page), `GET /login` / `GET /callback` / `POST /logout` / `GET /me` (Spotify login), `POST /scan` / `POST /scan/{id}/resume` (requires login; `GET /status/{id}` on a scan job additionally checks the requester's session owns it, since scan results are someone's private library contents), `POST /jobs` / `POST /jobs/batch` (capped at `MAX_BATCH_JOBS = 25`) / `GET /status/{id}` / `GET /download/{id}` (all anonymous, unchanged from before тАФ the manual download flow never required login). `/healthz` (Render's health check) is on the bare `FastAPI` app.
- `frontend/` тАФ React + TypeScript + Vite SPA. `src/api.ts` has typed fetch wrappers for every route above. Components: `ConnectAccount` (login/logout), `DownloadFolderPicker` (see below), `ScanPanel` (market input, progress incl. the `rate_limited`/Resume state, checkbox results list, "download selected" тЖТ `POST /jobs/batch`), `ManualDownloadForm` (the original single artist/album form), `DownloadsPanel` (polls an arbitrary list of download job ids, each row polling independently so adding a new job never disturbs existing pollers), `JobStatusView` (shared status/spinner/error rendering). `src/styles.css` carries the same dark/Spotify CSS custom properties (`--bg`, `--card`, `--green`, etc.) the pre-React reskin established тАФ a continuation, not a redesign.

**Local folder integration is entirely client-side, no server involvement.** `DownloadFolderPicker` + `src/downloadFolder.ts` use the browser's File System Access API (`window.showDirectoryPicker({ mode: "readwrite" })`) to let a friend point the app at their real local music folder тАФ one picker grant covers both features below, since readwrite subsumes read:
- **"Skip albums I already have":** the frontend reads the folder's existing subdirectory names directly (`FileSystemDirectoryHandle.entries()`) and cross-references them (via a JS port of `album_downloader.py`'s `sanitize_filename()`, kept in sync by hand тАФ see `downloadFolder.ts`'s `sanitizeFilename()`) against scan results and the manual download form, disabling/skipping anything already present. Chosen deliberately over a server-side "remember what this friend downloaded" history: this reads the *actual current folder contents* fresh every time, so it stays correct no matter how much time has passed or how many server restarts happened in between тАФ no persistence, no expiry, no staleness question at all.
- **Save straight to the folder, no zip step:** once a download job finishes, if a folder is chosen, `DownloadsPanel`'s `TrackedJobRow` automatically `fetch()`es the same zip the server has always produced (`GET /download/{job_id}` тАФ **the backend is completely unchanged**), unzips it client-side with `fflate` (`unzipSync`), and writes each file into an `<Artist> - <Album>` subfolder (`getDirectoryHandle(..., {create: true})` + `getFileHandle` + `createWritable()`) using the exact same naming (`albumFolderName()`) the duplicate-check already relies on. A manual "Save to folder" / "Retry save to folder" button is always available too, covering: a folder picked after the job already finished, or retrying a failed write (e.g. permission revoked mid-session). The server's zip is deliberately kept as-is rather than restructured to serve individual files тАФ it's still the fallback delivery mechanism for browsers without this API, and unzip-in-the-browser (`fflate`, ~8kB, zero deps) is far less invasive than reworking `downloader_adapter.py`'s zip-then-delete pipeline. Notably, that zip is **flat inside** (`shutil.make_archive(..., root_dir=dest_folder)` in `downloader_adapter.py` elides the "Artist - Album" wrapper), so the frontend synthesizes that folder name itself rather than trusting the zip's internal paths.

Both features degrade identically on unsupported browsers: `isDirectoryPickerSupported()` feature-detects Chromium-only `showDirectoryPicker` (Firefox/Safari don't implement it), and the picker component renders nothing when unsupported тАФ those friends just get today's original behavior (a plain "Download zip" link, no duplicate-skipping), not an error. Requires two frontend-only additions: the `@types/wicg-file-system-access` dev dependency, since TypeScript's bundled DOM lib doesn't include these types yet (must also be added to `tsconfig.app.json`'s `"types"` array, not just installed тАФ that array opts out of TypeScript's default "include every `@types/*` package" behavior, so an installed-but-unlisted types package is silently ignored, which is *not* caught by a plain `tsc --noEmit` in this project - only `tsc -b` per `npm run build`, since the root `tsconfig.json` is solution-style with an empty file list); and the `fflate` runtime dependency for client-side unzipping.

## Deployment

Render.com free "Web Service" tier, Docker-based, now a **multi-stage** build: a `node:22-slim` stage runs `npm ci && npm run build` for `web/frontend/`, then the final `python:3.12-slim` stage `COPY --from=`s just the built `dist/` тАФ the runtime image stays Node-free. Build output (`web/frontend/dist/`) is deliberately **not committed to git** (built fresh in the Dockerfile every deploy) to avoid a single-maintainer footgun: edit the frontend, forget to rebuild, silently deploy a stale UI. `web/frontend/node_modules/` and `dist/` are excluded via both `.gitignore` (frontend's own, nested) and `.dockerignore` (root) тАФ the latter matters specifically so a locally-built `node_modules` (Windows binaries) never gets copied over the Linux ones `npm ci` installs inside the build stage.

Rejected alternatives (frontend build): committing `dist/` directly and skipping the Node build stage entirely тАФ simpler and faster, but trades away the "can't forget to rebuild" safety net for a project with one maintainer.

**Manual step required, not automatable:** `SPOTIFY_WEB_REDIRECT_URI` (the web app's OAuth callback, e.g. `https://spotify-album-downloader.onrender.com/callback`) must be registered as an *additional* Redirect URI on the Spotify Developer Dashboard app тАФ separate from `sync_missing.py`'s own `SPOTIFY_REDIRECT_URI` (a CLI loopback address; don't conflate the two env vars). Spotify's Redirect URIs are dashboard-only, no API for it. Also set `SPOTIFY_WEB_REDIRECT_URI` as an env var both locally and on Render.

Render.com reasoning (Fly.io/Cloud Run/raw-VM tradeoffs, cold-start behavior) and the GitHub-App auto-deploy setup are unchanged from before тАФ see git history if needed.

## Known limitations, accepted rather than solved

- In-memory job **and session** state means an in-progress job, an in-progress scan, and every friend's login are all lost if the free host restarts/spins down. Logging back in is the only recovery. Both the per-user scan cache (`web/.web_scan_cache/*.json`) and the shared catalog cache (`web/.web_scan_cache/catalog.db`) are on the same ephemeral disk, so both reset on a spin-down/redeploy too тАФ they still help a lot within one warm instance's lifetime (repeated scans, auto/manual resumes, and now cross-friend reuse), but degrade to a cold scan afterward, same as before either cache existed. Making them survive a spin-down would need real persistent storage (a Render persistent disk or small external DB/KV store) тАФ not pursued, since it's added infra/cost for what's still a friends-only hobby project.
- Spotify rate limits (Client Credentials **and** the per-user Authorization Code flow, since scans use the latter) are shared app-wide across every user of the site тАФ a big library scan can burn through the same quota `album_downloader.py`'s lookups and other friends' scans share. Scans capped at `SCAN_WORKER_COUNT = 1` limits this app's own concurrency but doesn't prevent two friends' scans from tripping each other's 429s back-to-back. The per-user cache + shared catalog cache + auto-retry (see `jobs.py`/`scan_adapter.py`/`catalog_cache.py` above) make this far less disruptive than before тАФ a repeat scan barely touches the rate limit at all, a scan that overlaps with another friend's prior scan touches it even less, and most rate-limit hits now resolve themselves without a click тАФ but it isn't eliminated.
- No "already downloaded" filtering on scan results (see `scan_adapter.py` above) тАФ deliberately not solved now; a session-scoped filter would only cover a narrow case (this browser session, not a friend's actual local library) and wasn't judged worth the complexity yet.
- Ephemeral disk means zips are transient by design (TTL cleanup), not persistent storage.
- YouTube's bot-check on datacenter IPs (see the `extractor_args` note in the root `CLAUDE.md`'s `album_downloader.py` section) is mitigated, not eliminated.
